<task type="auto">
  <name>Task 1: Add highlight fields to RenderConfig</name>
  <files>src/codepicture/config/schema.py</files>
  <action>
    Add two new fields to the RenderConfig class, placed after the existing `background_color` field in a new "Highlighting" section comment:

    ```python
    # Highlighting
    highlight_lines: list[str] | None = None  # e.g. ["3", "7-12", "15"]
    highlight_color: str | None = None  # e.g. "#FFE65040"
    ```

    Add a field_validator for `highlight_color` (mode="before"):
    - If None, return None
    - If not a string, raise ValueError
    - Validate against regex `^#([0-9a-fA-F]{6}|[0-9a-fA-F]{8})$` -- only #RRGGBB and #RRGGBBAA (NOT #RGB, because 3-char hex is ambiguous for alpha)
    - Return the validated string

    Add a field_validator for `highlight_lines` (mode="before"):
    - If None, return None
    - If not a list, raise ValueError
    - For each spec in the list: validate format against regex `^\d+(-\d+)?$`
    - If any spec is invalid, raise ValueError with helpful message
    - Convert all specs to strings (in case TOML parsed as integers): return [str(s) for s in v]

    Important: Do NOT resolve line ranges in validators. Validators check FORMAT validity only. Resolution to 0-based indices happens at render time when total_lines is known. Store user intent as strings.

    TOML format is flat top-level fields (consistent with all other config fields):
    ```toml
    highlight_lines = ["3", "7-12", "15"]
    highlight_color = "#FFE65040"
    ```

    This works automatically because `load_config` passes the TOML dict to `RenderConfig.model_validate()`, and these field names match the TOML keys directly.
  </action>
  <verify>
    Run in Python:
    ```
    from codepicture.config.schema import RenderConfig
    # Valid cases

    assert c2.highlight_color == "#FF000040"

        pass
    try:

        assert False, "Should have raised"
    except Exception:
        pass
    ```
  </verify>
